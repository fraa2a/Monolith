# Decisions

This file is the architecture decision record index for the project.

## ADR-0001: Start from scratch instead of forking OBS Studio

- Status: accepted
- Decision:
  - Build a custom native recorder-first application rather than forking OBS Studio.
- Notes:
  - OBS remains a benchmark/reference point, not the product base.

## ADR-0002: Native Windows app instead of Electron/WebView

- Status: accepted
- Decision:
  - Use a native Windows desktop application architecture with no Electron or WebView runtime.

## ADR-0003: C++ and CMake as the initial implementation path

- Status: accepted
- Decision:
  - Use C++23 and CMake for the initial implementation.

## ADR-0004: FFmpeg/libav considered for the encoding backend

- Status: accepted
- Decision:
  - Prefer FFmpeg/libav for encoding and muxing unless early spikes force revision.

## ADR-0005: WASAPI for audio capture

- Status: accepted
- Decision:
  - Use WASAPI for loopback and microphone capture.

## ADR-0006: Win32 hotkeys and tray integration

- Status: accepted
- Decision:
  - Use `RegisterHotKey` for global shortcuts and `Shell_NotifyIcon` for tray integration.

## ADR-0007: Local IPC for Stream Deck plugin communication

- Status: accepted / resolved
- Decision:
  - v1 IPC transport = localhost WebSocket/TCP JSON-RPC (JSON-RPC over TCP).
  - Named pipes deferred; rationale preserved for future evaluation.
- Notes:
  - host: 127.0.0.1, port: 45991 (matches default-config.json).

## ADR-0008: Single-process MVP with modular internals

- Status: accepted
- Decision:
  - Build the MVP as a single executable (app/recorder) with strict internal module
    boundaries enforced by /libs separation. Two-process split is the future target
    once the MVP is stable.
- Notes:
  - Deferred rationale: crash isolation and independent restart remain the long-term goal.

## ADR-0009: Settings storage — SQLite (`settings.db`), UI is the Deno popup

- Status: **superseded** (originally: WinUI 3 sidecar + `config.json`; rewritten in
  the F5 UI migration).
- Original decision (historical): per-user config at `AppData\Local\Monolith\config.json`
  via `nlohmann-json` merged over `config/default-config.json`, edited by a WinUI 3
  sidecar that reloaded on exit.
- Current decision:
  - The per-user settings store of record is **`AppData\Local\Monolith\settings.db`**
    (SQLite, WAL), managed by `libs/storage` as a generic key/value table with one
    row per top-level config section. `config.json` is **retired**; on first run the
    engine imports a legacy `config.json` once (backing it up as
    `config.json.imported.bak`) and seeds defaults into `settings.db`.
  - The engine reads `settings.db` directly in `settings_config.cpp` (same
    merge-over-defaults + validate/clamp logic as before; `config/default-config.json`
    remains the default seed / compiled fallback). The Deno UI settings popup is the
    writer: it writes `settings.db` and calls the `reload_settings` JSON-RPC so the
    engine re-reads and re-applies live/deferred scopes.
  - Settings UI moves from WinUI pages to a **custom popup/overlay** in the Deno app
    (General / Output / Clip / Capture / Audio / Hotkeys / Advanced / Game Detection),
    capability-gated by `runtime-status.json`.
- Notes:
  - Reload no longer keys off sidecar process exit (the Deno UI is a persistent
    window); it is an explicit `reload_settings` IPC call after Save. See ADR-0011.
  - Hotkey fields are now editable in the popup (previously read-only).

## ADR-0010: SQLite clip catalog co-located per output folder

- Status: accepted
- Context:
  - The UI is being rebuilt around a clip library (Home = clip grid). Bare video
    files with no metadata cannot back a searchable/filterable library, favorites,
    hashtags, or per-clip game association.
- Decision:
  - Add a per-output-folder SQLite catalog: `<folder>\clips.db` (replay) /
    `<folder>\recs.db` (manual), co-located with the video files, plus first-frame
    PNG thumbnails under `<folder>\.thumbs\`.
  - Native dependency = vcpkg `sqlite3` (pinned baseline). New `libs/storage`
    module owns schema, WAL open, insert, mutations, and self-heal/reconcile.
  - The always-on recorder is the single writer (at clip-save and reconcile, and —
    from a later phase — for UI-driven favorite/hashtag/delete over IPC). The Deno
    UI opens the same DB read-only for the grid. All connections use WAL +
    `busy_timeout` so reads never block the writer.
  - Thumbnails are generated by the engine via `encoding::generate_thumbnail`
    (libav first-frame decode → scaled PNG), off the UI/tray thread.
  - Self-heal on startup: a row whose video is gone is deleted (as if the clip was
    removed); a missing thumbnail is regenerated; pre-existing videos with no row
    are imported (migration). The UI additionally shows a placeholder when a
    thumbnail is missing/errored and still starts the mini-player.
- Notes:
  - Layout keeps the video files where they already live inside the chosen output
    folder (DB + `.thumbs` added alongside), rather than nesting an extra `Clips/`
    subfolder — the folder is still self-contained, with less disruption to the
    existing write path. A non-Monolith / wrong-schema `.db` is never overwritten;
    it is reported (same spirit as the `config.json.bak` guard).
  - The settings-store move (config.json → SQLite) and the Deno/webview UI
    exception each get their own ADR when those phases land.

## ADR-0011: UI layer moves to Deno Desktop (WebView2) — engine stays native

- Status: superseded by ADR-0012 (shell runtime only). The UI-exception,
  native-engine, and JSON-RPC-contract decisions below still hold; only the
  **Deno Desktop** shell/runtime was replaced by Tauri.
- Context:
  - The UI is being rebuilt as the heart of the app (Home = clip grid, Settings =
    popup) per `deno.md`. WinUI 3 is heavy and slow to iterate; a web UI is the
    right tool for a data-driven media library. This conflicts with ADR-0002 /
    DEVELOPMENT_RULES rule 2 (no WebView).
- Decision:
  - Restrict the no-webview rule to the capture/recording **engine**. The **UI
    layer only** (`app/desktop-ui`) may use a system WebView2 window. The engine
    (`app/recorder` + `libs/*`) remains 100% native (no Electron, no CEF, no
    browser runtime), preserving the spirit of ADR-0002 where it matters.
  - Implementation: Deno Desktop app compiled to a self-contained `Monolith.UI.exe`
    via `deno compile`, dropped next to `Monolith.exe` under `ui/` and launched
    from the tray (settings_window.cpp), preferred over the legacy WinUI sidecar
    during migration.
  - Webview shell = **Deno Desktop** (`deno desktop`, Deno 2.9+), backend
    `webview` (WebView2 on Windows). `deno desktop` compiles the app to a
    self-contained **bundle directory** `Monolith.UI/` (launcher
    `laufey_webview.exe` + `Monolith.UI.dll`), run as
    `laufey_webview.exe --runtime Monolith.UI.dll`. Frontend = Preact + esbuild
    (Deno loader), bundled into `dist/` and embedded via `--include dist`.
  - Deno Desktop auto-binds the window to `Deno.serve()`, so there is no manual
    webview/port wiring: `main.ts` just serves. The server exposes the built
    frontend, a JSON API over the **read-only** clip catalogs, and range-streamed
    media/thumbnails. It talks to the engine only over the existing JSON-RPC
    (127.0.0.1:45991).
  - `@db/sqlite` loads a native `sqlite3.dll` (downloaded+cached on first run, or
    shipped next to the launcher with `DENO_SQLITE_PATH` set for a fully offline
    app — `main.ts` sets this automatically when the dll is present).
  - Single-writer rule: UI-driven clip mutations (favorite/hashtag/delete) are
    **not** written by the UI; they are sent to the engine over new IPC methods
    (`clip_set_favorite`, `clip_add_hashtag`, `clip_remove_hashtag`, `clip_delete`)
    so the recorder stays the sole writer of `clips.db`/`recs.db`. `reload_settings`
    IPC replaces the old settings-process-exit reload trigger.
- Notes:
  - The legacy WinUI project is retained as `app/OLD-settings-ui` and keeps
    building until the Deno UI reaches parity (deno.md §1), then is removed.
  - Config still lives in `config.json` for now; the settings→SQLite move (which
    also rewrites ADR-0009) lands in a later phase.

## ADR-0012: Desktop UI shell moves from Deno Desktop to Tauri (WebView2)

- Status: accepted (supersedes the shell/runtime implementation of ADR-0011)
- Context:
  - `deno desktop` is experimental and failed the most basic acceptance
    criterion: the compiled/run UI process could exist with **no visible
    window**. Adding startup logs + `BrowserWindow.show()`/`focus()` did not fix
    it. Monolith needs a reliable Windows-only sidecar window launched from the
    tray, so the shell/runtime — and only the shell/runtime — is replaced.
  - Everything ADR-0011 decided about *scope* stands: the WebView exception is
    still limited to `app/desktop-ui`; the engine (`app/recorder` + `libs/*`)
    stays 100% native; the engine remains the single writer of `clips.db`/
    `recs.db`; UI↔engine still goes over JSON-RPC on `127.0.0.1:45991`.
- Decision:
  - Shell = **Tauri v2 / WebView2** under `app/desktop-ui/src-tauri/`. `cargo
    build --release` produces a single self-contained `Monolith.UI.exe`
    (`monolith_ui.exe`, copied to `<recorder-output>/ui/Monolith.UI.exe`). The
    tauri-cli is **not** required — `bundle.active=false`; we ship the bare exe.
  - The **Preact frontend is unchanged**. Rather than rewrite every component to
    Tauri `invoke()` calls, the Rust host runs a small **loopback HTTP server**
    (`tiny_http`, `server.rs` + `media.rs`) that preserves the exact same-origin
    route contract the frontend already uses (`/api/*`, `/media/*`, `/thumb/*`,
    static assets, with `<video>` Range support). The window is created in Rust
    (`WebviewWindowBuilder` → `WebviewUrl::External("http://127.0.0.1:<port>/")`)
    against that server, which guarantees a visible window.
  - Backend behaviour is ported to focused Rust modules mirroring the old Deno
    `bindings/`: `paths.rs`, `settings_store.rs` (reads/writes
    `settings.db`), `clip_catalog.rs` (read-only `clips.db`/`recs.db`),
    `engine_rpc.rs` (JSON-RPC), `game_catalog.rs`. SQLite is `rusqlite` with the
    `bundled` feature — **no separate `sqlite3.dll` to ship** (this removes the
    Deno `@db/sqlite` first-run DLL download entirely).
  - The frontend is still bundled by `build.ts` (Deno + esbuild → `dist/`); the
    exe embeds `dist/` at compile time via `include_dir!`, so it is fully
    self-contained. CMake requires both `deno` (frontend bundle) and `cargo`
    (host); if either is absent it warns and skips the UI, non-fatal, matching
    the prior optional-toolchain behaviour.
- Notes:
  - Deferred to a later slice: background weekly refresh of the Discord game DB
    — `game-icon`/`game-catalog` serve whatever snapshot exists and return
    `null`/empty otherwise. (The old `scripts/download-game-db.ts` was removed;
    the refresh will be re-implemented in Rust when needed.)
  - Cleanup done (follow-up to this ADR): `app/OLD-settings-ui` (the WinUI/.NET
    settings sidecar) and the old Deno host files (`main.ts`, `server/`,
    `bindings/`, `scripts/`, `deno.lock`) were deleted. The recorder no longer
    falls back to a legacy UI, the installer ships `ui/Monolith.UI.exe` instead
    of `settings/`, and CI dropped the .NET toolchain for Deno + Rust. Deno is
    retained as the build-only frontend bundler (not shipped).
